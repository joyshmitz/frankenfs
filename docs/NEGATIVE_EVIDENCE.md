### 2026-06-22 parallel-gap lever-map: distributed per-read overhead × shared-cacheline contention (not one lock) (CrimsonFox cc/opus)

Profiled the parallel ext4 single-file random read (`frag.bin`, 200k×4 KiB, 64t) to pinpoint the ~10x coordination gap localized in the prior synthesis. Self-time breakdown:
- **`pread` syscall ~45%** (`__libc_pread` 12% + `read_exact_at` 13% + syscall wrapper) — one pread per 4 KiB read, same count the kernel issues, but frankenfs wraps each in far more userspace work.
- **`__sched_yield` ~14% + parking_lot `lock_slow`/`unlock_slow` ~16%** (`WordLock` 7.1% + `RawMutex` 6.0% + unlock 2.7%) — threads parking on contended locks.
- per-read inode RE-READ + RE-PARSE: `read_inode_with_scope`/`read_inode_raw_with_scope`/`parse_from_bytes_with_ibody`/`read_file_data` (~4% combined) → each hits the inode-table-block `ShardedCache` Mutex for the SAME hot inode block.
- `ExtentCache::lookup` 3.3% — already optimal on HITS (read-lock + atomic `fetch_max` for LRU, no write lock), BUT a single-file read = one namespace = ONE shard, so 64 threads ping-pong that shard's `RwLock` reader-count cacheline (the bd-tag2s effect, inherent to per-namespace sharding for single-file workloads).
- MVCC adapter create+drop per read (`block_device_adapter` + `drop_glue::<MvccBlockDevice>` ~1.8%).

ROOT CHARACTERIZATION: the parallel gap is NOT one contended lock — it is **distributed per-read overhead (inode re-parse, extent lookup, MVCC adapter churn, multiple cache-lock acquires) multiplied by 64-thread shared-cacheline contention** on the two single-file hot spots (the namespace's `ExtentCache` RwLock reader-count + the inode block's `ShardedCache` Mutex). Each piece is 1-4% self; the aggregate + the cacheline ping-pong is the 10x. No single ≥1.1x surgical lever; the real wins would be (a) a per-inode PARSED-inode RO cache to kill the per-read re-parse + one lock acquire, and (b) reducing per-read lock acquisitions on single-file workloads — both deeper changes in the shared read dispatch (bd-xmh5g.414/.382 coordination family, peer-active). Documented as a lever-map for that work; no code this turn (no cheap safe single lever).

### 2026-06-22 scrub is ALLOCATION-bound, not validation-bound — parallelizing validation would NOT help (CrimsonFox cc/opus)

Investigated `ffs-cli scrub` as a read-only perf axis (256 MiB btrfs image, warm, ~63-73 ms). The serial per-block validation loop (`scrub_range`, ffs-repair/src/scrub.rs) LOOKS like an obvious parallelization target (each `validator.validate(block, buf)` is independent, `BlockValidator: Send+Sync`, rayon already a dep). perf REFUTES that intuition — self-time is dominated by **buffer allocation/zeroing**, not validation:
- `read_exact_at`/`pread` ~26% (unavoidable data movement)
- `__memset_avx2` ~22% + `AlignedVec::new`/`calloc` ~19% + `mprotect`/`madvise` ~21% + `Arc<AlignedVec>::drop_slow`/`_int_free` ~14% (per-batch buffer alloc/zero/free churn — large allocs go through mmap → mprotect/madvise)
- `ZeroCheckValidator::validate` only **7.3% self**; csum validators negligible (only fsid-matching candidate blocks)

So parallelizing the validation loop would chase a 7% slice and ADD rayon overhead on 64-block batches — net ~inert (same trap as the reverted single-job-inline lever). The real scrub lever is **buffer reuse**: `ByteBlockDevice::read_contiguous_blocks` (ffs-block:1189) allocates+zeroes a `vec![0; batch_len*block_size]` (~1 MiB) staging buffer per batch on its non-fast path, and the `BlockBuf` pool's `AlignedVec::from_vec` realigns per fill. The fast vectored path (no staging) IS available (`FileByteDevice::preserves_read_vectored_destinations_on_error()==true`) but the alloc churn persists across batches per the profile. DEFERRED: the fix is in ffs-block's shared read-staging path (high blast radius — used by every read, not just scrub) for a modest win on a rarely-run op; not a cheap safe single-crate lever. Recorded so the "parallelize scrub validation" intuition is not re-chased — the bottleneck is allocation, and the safe place to fix it is the staging-buffer lifecycle, not the validation loop.

### 2026-06-22 SHIPPED: scrub chunk-size fix — buffer-pool reuse, ~1.06-1.22x (CrimsonFox cc/opus)
